# Разработка Karyolab v2

Этот раздел собран **для инженеров**: дизайн, фронтенд, бэкенд, инфраструктура. Здесь живёт документация, на основе которой пишется код.

Концептуальные документы для учёного-заказчика лежат отдельно в [`docs/концепция/`](../концепция/README.md). Если научное правило в концепции и в разработке расходятся — **истина в концепции**.

## Структура Раздела

- [`дизайн/`](дизайн/README.md) — визуальные брифы и UX-правила по разделам. Опираются на макеты заказчика.
- [`фронтенд/`](фронтенд/README.md) — стек, навигация, локализация, контракты с бэкендом.
- [`бэкенд/`](бэкенд/README.md) — архитектура, модель данных, API, хранение файлов, аутентификация, бэкапы.

## Принципы Разработки

1. **Концепция — главный источник правды**. Перед реализацией функции инженер обязан прочитать соответствующий концептуальный документ. Если в концепции дыра — задаём вопрос, не додумываем.
2. **Никаких хардкодных списков**, которые в концепции живут как справочники атласа (типы аномалий, классы хромосом, субгеномы, виды). Всё это должно быть редактируемо оператором через UI.
3. **Сохранение происхождения**. Каждая хромосома, идеограмма, кариотип знают свою цепочку до образца. На уровне БД это первичные ключи и foreign keys, на уровне API — обязательные поля `source_*`.
4. **Иммутабельные канонические ID**. ID хромосомы (`<метафаза>.cNN`) не меняется. Имя файла — отдельная характеристика.
5. **Кариотип образца хранит копии файлов хромосом**. Это решение модели данных: при сохранении кариотипа образца файлы выбранных хромосом копируются в его папку.
6. **Никакой автоматической биологии**. Программа предлагает разметку, но не принимает решения за оператора. Особенно — не отмечает автоматически трисомии, моносомии, замещения.
