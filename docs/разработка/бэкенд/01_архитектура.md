# Архитектура Бэкенда

## Слоистая Структура

```
┌─────────────────────────────────┐
│  HTTP / REST (Fastify)          │  ← запросы фронта
└──────────────┬──────────────────┘
               │
┌──────────────▼──────────────────┐
│  Контроллеры (handlers)         │  ← валидация запроса, авторизация
└──────────────┬──────────────────┘
               │
┌──────────────▼──────────────────┐
│  Сервисы (бизнес-логика)        │  ← state machine'ы, оркестрация
└──────────────┬──────────────────┘
               │
┌──────────────┴──────────────────┐
│  Репозитории / ORM (Prisma)     │  ← доступ к БД
└──────────────┬──────────────────┘
               │
┌──────────────▼──────────────────┐
│  PostgreSQL + файловое хранилище│
└─────────────────────────────────┘
```

Параллельно:

- **Очередь задач** (BullMQ + Redis) для тяжёлых операций: парсинг PSD, генерация TIFF-обзоров.
- **Аудит-лог** пишется внутри сервисов одной транзакцией с основной операцией.

## Модули

Бэкенд разбит на **домены**, повторяющие концепцию:

```
src/
├── modules/
│   ├── journal/          # образцы, ивенты, статусы, прогресс, тильт
│   ├── karyotype/        # импорт PSD, хромосомы, идеограммы, кариотипы, экспорт
│   ├── atlas/            # справочники, сравнения, эталоны
│   ├── files/            # работа с файловым хранилищем
│   └── auth/             # минимальная авторизация (если нужна)
├── shared/
│   ├── ids.ts            # генерация и валидация иерархических ID
│   ├── audit.ts          # запись аудит-лога
│   ├── errors.ts         # типизированные ошибки
│   └── db.ts             # клиент Prisma
├── workers/
│   ├── psd-parser.ts     # обработка PSD из очереди
│   └── export-builder.ts # генерация TIFF/Excel из очереди
└── server.ts             # запуск Fastify
```

Каждый модуль содержит:

- `routes.ts` — регистрация HTTP-эндпоинтов;
- `service.ts` — бизнес-логика;
- `repository.ts` — доступ к БД через Prisma;
- `schemas.ts` — Zod-схемы запросов и ответов;
- `types.ts` — внутренние типы модуля.

## Технологии (Закреплено)

| Слой | Выбор | Обоснование |
|---|---|---|
| Язык | TypeScript (Node.js 20+) | общий язык с фронтом, минимум контекст-свитча |
| HTTP-фреймворк | Fastify | быстрый, JSON-схемы из коробки, Pino-логи |
| ORM | Prisma | строгая типизация, миграции, удобный DX |
| БД | PostgreSQL 15+ | проверенный стек, JSON-поля для гибких атрибутов |
| Очередь | BullMQ + Redis | для тяжёлых операций (PSD, экспорт TIFF) |
| Валидация | Zod | общий для фронта и бэка тип-сейф |
| Логи | Pino + ротация | стандартный для Fastify |
| Тесты | Vitest | быстрый, совместимый с Vite на фронте |
| Линт | ESLint + Prettier | как у фронта |

## Безопасность Транзакций

Любая операция, которая меняет несколько таблиц, идёт **в одной транзакции**. Примеры:

- Утверждение кариотипа образца: запись в `sample_karyotypes` + копирование файлов хромосом + запись в `audit_log` + смена статуса образца → одна транзакция (с компенсацией для копирования файлов).
- Замена хромосомы в утверждённом кариотипе: копирование нового файла + обновление `stored_chromosomes` + запись в `audit_log` → одна транзакция.

Файловые операции, которые нельзя откатить (копирование больших PSD), идут **до** записи в БД, и в случае ошибки БД временные файлы удаляются.

## Идемпотентность

Тяжёлые операции (импорт PSD, генерация TIFF) запускаются с **idempotency key** от клиента. Повторный запрос с тем же ключом возвращает уже посчитанный результат, а не запускает повторно.

## Версионирование API

`/api/v1/...`. Если в будущем потребуется ломающее изменение — `/api/v2/...` параллельно. Внутренние клиенты (наш фронт) переходят согласованно.

## Конфигурация

Все параметры через `.env`:

```
DATABASE_URL=postgresql://...
REDIS_URL=redis://...
MEDIA_ROOT=/var/karyolab/media
S3_ENDPOINT=...
S3_BUCKET=karyolab-media
S3_ACCESS_KEY=...
S3_SECRET_KEY=...
PORT=3000
LOG_LEVEL=info
```

Чувствительные значения (`S3_SECRET_KEY`, `DATABASE_URL`) приходят от Railway или из секрет-стора, не лежат в репозитории.
