# Бэкапы И Восстановление

Karyolab v2 хранит **очень ценные данные** — годы лабораторной работы, размеченные кариотипы, фотографии. Потеря этих данных недопустима. Бэкапы — обязательная часть продакшен-установки.

## Что Бэкапим

| Слой | Что | Частота |
|---|---|---|
| Реляционная БД | `pg_dump` всей БД | ежедневно (полный), плюс WAL-archiving для PITR |
| Файловое хранилище | `MEDIA_ROOT/` или S3-бакет | ежедневно (инкрементальный) |
| Конфиги (`.env`, `prisma/migrations/`) | в репозитории | по коммиту |
| Аудит-лог | вместе с БД | вместе с БД |

## Стратегия

### Локальная Установка

Если приложение работает локально на компьютере лаборатории:

- ежедневный `pg_dump` в `$BACKUP_ROOT/db/<YYYY-MM-DD>.sql.gz`;
- ежедневный `rsync` или `restic` снимок `$MEDIA_ROOT/` в `$BACKUP_ROOT/media/`;
- ротация: 7 ежедневных + 4 еженедельных + 12 ежемесячных;
- **обязательная внешняя копия** на отдельный диск или облако (минимум раз в неделю).

Скрипт `scripts/backup-local.sh`:

```bash
#!/bin/bash
set -euo pipefail
DATE=$(date +%Y-%m-%d)
mkdir -p "$BACKUP_ROOT/db" "$BACKUP_ROOT/media"
pg_dump "$DATABASE_URL" | gzip > "$BACKUP_ROOT/db/$DATE.sql.gz"
restic -r "$BACKUP_ROOT/media" backup "$MEDIA_ROOT"
```

Скрипт ставится в cron на ночь.

### Облачная Установка (Railway / VPS)

- Postgres-провайдер делает ежедневные снапшоты + WAL-archiving (PITR).
- Медиа в S3-совместимом хранилище — версионирование бакета включено.
- Дополнительно — еженедельная репликация в **другой регион**.

## Восстановление

### Полное Восстановление

1. Создать пустую БД нужной версии.
2. `gunzip < backup.sql.gz | psql $DATABASE_URL`.
3. Восстановить `MEDIA_ROOT/` из `restic` или скопировать содержимое S3-бакета.
4. Запустить приложение, проверить, что превью открываются и кариотипы целы.

Время восстановления: до **1 часа** для базы порядка 5-10 ГБ.

### Восстановление Одного Кариотипа

Если оператор случайно удалил кариотип образца:

1. Soft-delete уже защищает: запись в `sample_karyotypes` не удалена физически, помечен `deleted_at`.
2. Чтобы восстановить — `UPDATE sample_karyotypes SET deleted_at = NULL WHERE id = ...`.
3. Файлы тоже не удалены (см. [04_хранение_файлов.md](04_хранение_файлов.md)).

Если оператор сделал **жёсткое удаление** через CLI — восстанавливаем из последнего бэкапа.

### Восстановление Идеограммы После Замены

История замен хромосом сохраняется (`<storage_dir>/chromosomes/_history/`). Если новая хромосома оказалась хуже старой:

1. Найти файл в `_history/`.
2. Использовать `PATCH /sample-karyotypes/:id/chromosomes` с тем же файлом.
3. Это создаст новую запись в `audit_log`, но утверждение не отзовётся.

## Тестирование Бэкапов

**Минимум раз в квартал** — фактическая проверка восстановления:

1. Берём свежий бэкап.
2. Поднимаем приложение в `staging` окружении.
3. Заходим во фронт, открываем 5-10 случайных образцов и кариотипов.
4. Проверяем, что превью грузятся, экспорт работает.

Без тестирования бэкапы — это вера. С тестированием — доказательство.

## Что Бэкапить НЕ Нужно

- `tmp/imports/` — временные файлы PSD, безопасно теряются.
- Кеш-превью — пересоздаются по требованию.
- Очереди (Redis) — пустеют между запусками, состояние задач можно восстановить из БД.
