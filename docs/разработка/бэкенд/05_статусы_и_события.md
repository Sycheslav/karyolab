# Статусы И События

В Karyolab v2 много state machine'ов: образец, препарат, окрашенный препарат, кариотип, хромосома. Этот документ описывает их реализацию на бэке.

## Подход

- Каждый state machine описан как **TypeScript enum + таблица переходов**.
- Сервис проверяет переход перед записью в БД. Если переход недопустим — возвращает 409.
- Любой переход пишется в `audit_log` с указанием `from_status` и `to_status`.

## Препарат

```ts
type PreparationStatus =
  | "создан"
  | "отмыт"
  | "гибридизован"
  | "сфотографирован, судьба не решена"
  | "переотмыт"
  | "выброшен";

const transitions: Record<PreparationStatus, PreparationStatus[]> = {
  "создан":                              ["отмыт", "выброшен"],
  "отмыт":                               ["гибридизован", "выброшен"],
  "гибридизован":                        ["сфотографирован, судьба не решена", "переотмыт", "выброшен"],
  "сфотографирован, судьба не решена":   ["переотмыт", "выброшен"],
  "переотмыт":                           ["гибридизован", "выброшен"],
  "выброшен":                            [],
};
```

Что вызывает переход:

| Переход | Триггер |
|---|---|
| `создан` → `отмыт` | ивент `wash` с этим препаратом |
| `отмыт` → `гибридизован` | ивент `hybridization` с этим препаратом |
| `гибридизован` → `сфотографирован, судьба не решена` | ивент `photography` с решением `deferred` |
| `гибридизован` → `переотмыт` | ивент `photography` с решением `rewashed` сразу |
| `гибридизован` → `выброшен` | ивент `photography` с решением `discarded` сразу |
| `сфотографирован, судьба не решена` → `переотмыт` | решение `rewashed` (через `POST /preparations/:id/decision` или сразу в `photography`) |
| `сфотографирован, судьба не решена` → `выброшен` | решение `discarded` |

> Постгибридизационная отмывка — **это решение**, не отдельный ивент.
> Статус `переотмыт` уже означает "готов к следующей гибридизации"; дополнительный ивент `wash` для возврата в `отмыт` не создаётся.

## Окрашенный Препарат

```ts
type StainedStatus =
  | "создан"
  | "сфотографирован"
  | "сфотографирован, судьба не решена"
  | "закрыт переотмывкой"
  | "закрыт выбрасыванием";
```

| Переход | Триггер |
|---|---|
| `создан` → `сфотографирован` | ивент `photography`, если судьба решена сразу |
| `создан` → `сфотографирован, судьба не решена` | ивент `photography` с решением `deferred` |
| `создан` → `закрыт переотмывкой` | ивент `photography` с решением `rewashed` сразу |
| `создан` → `закрыт выбрасыванием` | ивент `photography` с решением `discarded` сразу |
| `сфотографирован, судьба не решена` → `закрыт переотмывкой` | позднее решение `rewashed` |
| `сфотографирован, судьба не решена` → `закрыт выбрасыванием` | позднее решение `discarded` |

`закрыт переотмывкой` означает, что физический препарат снова готов к гибридизации, а текущий окрашенный препарат закрыт как завершённый цикл.

## Кариотип Образца

```ts
type SampleKaryotypeStatus = "черновик" | "утверждён" | "экспортирован" | "архивный";

const transitions = {
  "черновик":      ["утверждён", "архивный"],
  "утверждён":     ["экспортирован", "архивный", "черновик"],
  "экспортирован": ["архивный", "утверждён"],
  "архивный":      ["черновик"],
};
```

| Переход | Триггер |
|---|---|
| `черновик` → `утверждён` | `POST /sample-karyotypes/:id/approve` |
| `утверждён` → `экспортирован` | автоматически при первом успешном экспорте |
| `утверждён` → `черновик` | `PATCH /sample-karyotypes/:id` с `status: "черновик"` (требует подтверждения от клиента) |
| `→ архивный` | `POST /sample-karyotypes/:id/archive` |

При утверждении дополнительно сервер проверяет, что у образца обновляется статус `есть результат` — если до этого не было.

## Хромосома

```ts
type ChromosomeStatus =
  | "новая"
  | "класс определён"
  | "есть идеограмма"
  | "кандидат в кариотип"
  | "выбрана";
```

`класс определён` и `есть идеограмма` могут сосуществовать — это не строго упорядоченные статусы, а флаги, которые сервис может объединить:

```ts
function deriveChromosomeStatus(c: Chromosome, ideo: Ideogram | null) {
  if (c.usedInSampleKaryotypeId) return "выбрана";
  if (c.manualCandidateForKaryotype) return "кандидат в кариотип";
  if (ideo)                      return "есть идеограмма";
  if (c.classCode)               return "класс определён";
  return "новая";
}
```

То есть фактический статус **вычисляется** из связей и ручных пометок оператора, а не хранится как независимая истина. `manualCandidateForKaryotype` не выставляется автоматически — это только ручной флаг. Колонка `status` в таблице — кеш для удобства фильтрации. Обновляется триггером или сервисным методом `recomputeChromosomeStatus`.

## Образец

| Статус | Условие |
|---|---|
| `зарегистрирован` | образец заведён вручную |
| `проращивается` | есть ивент `germination`, нет растений |
| `в работе` | есть растения, препараты, отмывки, гибридизации или фотографии, но нет утверждённого кариотипа |
| `есть результат` | есть утверждённый `sample_karyotype` |
| `архивный` | вручную |

Тоже **вычисляется** периодически (раз в день или при триггерных изменениях), кешируется в `samples.status`.

## Процессоры Ивентов

Каждый тип ивента обрабатывается отдельным сервисом. Контракт:

```ts
interface EventProcessor<TPayload> {
  validate(payload: TPayload): void;          // бросает ValidationError
  apply(payload: TPayload, tx: Transaction): Promise<EventApplyResult>;
  describe(payload: TPayload): EventSummary;  // для карточки ивента
}
```

`apply` внутри транзакции:

1. Создаёт запись в `events`.
2. Создаёт `event_participants`.
3. Меняет статусы участников.
4. Создаёт сопутствующие объекты (например, `stained_preparations` при `hybridization`).
5. Пишет `audit_log`.

## Прогресс

`GET /journal/progress` — отдельный сервис, который пакетно считает пять колонок. Колонки прогресса не являются тем же самым, что верхнеуровневый `samples.status`: например, `созрели` вычисляется по таймлайну проращивания и отсутствию препаратов.

```sql
-- созрели: образцы, по которым завершён таймлайн проращивания,
-- но ещё нет препаратов
SELECT s.* FROM samples s
WHERE s.deleted_at IS NULL
  AND EXISTS (... завершённый germination ...)
  AND NOT EXISTS (... preparations для sample ...);

-- ждёт отмывку
SELECT * FROM preparations
WHERE status = 'создан'
  AND deleted_at IS NULL;

-- отмыт
SELECT * FROM preparations
WHERE status IN ('отмыт', 'переотмыт')
  AND deleted_at IS NULL;

-- гибридизован
SELECT sp.* FROM stained_preparations sp
WHERE sp.status = 'создан'
  AND NOT EXISTS (SELECT 1 FROM events e
                  WHERE e.type = 'photography'
                    AND ...);

-- есть результат
SELECT s.* FROM samples s
WHERE s.status = 'есть результат';
```

Все, кроме "есть результат", — без пагинации (полный список). "Есть результат" — пагинированный.
