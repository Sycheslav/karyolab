# Миграция Данных

Заказчик использует первую версию Karyolab. В v2 модель данных изменилась: появились разделение между кариотипом метафазы и кариотипом образца, два уровня аномалий, копии хромосом в кариотипе, новые статусы. Часть данных может потребоваться перенести.

## Решение По Миграции

На момент написания **полной автоматической миграции из v1 пока не планируется**. Заказчик предположил, что некоторые ключевые материалы будут перевнесены вручную, потому что:

- в v1 разметка велась по другой логике;
- многие записи в v1 — черновые и не нуждаются в переносе;
- в v2 хочется сразу пользоваться новыми справочниками и аккуратной иерархией.

Тем не менее на случай решения мигрировать — проектируем миграционный модуль с минимальной поверхностью.

## Сценарии Миграции

### Полная Миграция

Если потребуется — отдельный CLI-инструмент `karyolab migrate-from-v1`:

1. Читает дамп БД v1 и медиа-файлы.
2. Маппит сущности v1 на сущности v2 по таблицам соответствия (см. ниже).
3. Создаёт записи в v2 с **новыми каноническими ID**, сохраняя старые ID в поле `legacy_id`.
4. Копирует медиа в новую структуру папок.
5. Логирует все несоответствия в файл отчёта.

### Частичная Миграция (По Образцу)

CLI `karyolab migrate-sample <sample_id>`:

- мигрирует один образец и связанные с ним растения, препараты, кариотипы;
- удобно для постепенного переноса.

## Таблица Соответствий V1 → V2

| V1 | V2 | Замечания |
|---|---|---|
| `Sample` | `samples` | имя, вид, родители — без изменений |
| `Slide` | `preparations` | статусы переразмаплены: `создан`/`отмыт`/... |
| `Hybridization` | ивент `hybridization` + `stained_preparations` | в v1 было одной сущностью, в v2 разделено |
| `Photo` | прикрепляется к `stained_preparations.photo_path` | |
| `KaryotypeOverview` | `sample_karyotypes` (с `is_reference` если был эталоном) | |
| `Reference E1/E2` | `sample_karyotypes` с `is_reference: true` | старая логика "один на класс на панель" не сохраняется — становятся обычными эталонами |
| `Anomaly (универсальный)` | разделяется на `chromosome_anomalies` или `genome_anomalies` | по контексту привязки |
| `ProbeSet` | `probes` + `stained_preparation_probes` | |

## Что Не Переносится Автоматически

- **Кастомные шаблоны экспорта v1** — в v2 модель шаблонов другая, переносим вручную.
- **Сравнения, которые жили во временных вкладках v1** — переоткрываются и сохраняются как `saved_comparisons` вручную.
- **Произвольные комментарии и заметки в полях, которые в v2 стали структурированными** — переносим в общий блок `notes` образца.

## Поле `legacy_id`

Каждой мигрированной записи добавляется поле `legacy_id` (если ещё не добавлено в схеме). Это позволяет:

- отследить происхождение,
- запустить миграцию повторно (повторный запуск находит существующую запись по `legacy_id` и обновляет, а не создаёт дубликат).

`legacy_id` — это `varchar(64) NULL`, индексируется отдельно.

## Тестирование Миграции

Перед финальной миграцией:

1. Берём анонимизированный дамп v1 от заказчика.
2. Запускаем миграцию в `staging`.
3. Проверяем, что 5-10 случайных образцов открываются в v2 и выглядят корректно.
4. Сверяем число объектов: сколько образцов, кариотипов, аномалий до и после.
5. Согласуем отчёт с заказчиком.
6. Только потом — миграция боевой базы.
