## Главные Принципы

### 1. Концепция — Источник Правды

Перед реализацией функции инженер открывает соответствующий документ в [`docs/концепция/`](../../концепция/README.md). Если в концепции не описано — либо пишем уточняющий документ, либо задаём вопрос заказчику. Не додумываем.

### 2. Каноничный ID Хромосомы Иммутабелен

ID `<метафаза>.cNN` присваивается при импорте PSD и **не меняется**. Имя файла — отдельная характеристика. На уровне БД это первичный ключ.

### 3. Кариотип Образца — Снимок

При сохранении кариотипа образца сервер **копирует файлы хромосом** в его папку. Это:

- защищает кариотип от изменений в исходных метафазах;
- позволяет менять хромосомы в утверждённом кариотипе без сброса утверждения.

### 4. Никаких Хардкодных Списков

Типы аномалий, классы хромосом, шаблоны субгеномов, виды, зонды — всё в БД, всё редактируется через API. На старте приложения — пустые справочники или минимальный шаблон, который оператор может заменить.

### 5. Иерархические ID

ID объектов — структурные строки вида `1730.25.1.1.1.m1.c01`. На уровне БД это `varchar` колонка, обычно индексированная для префиксного поиска. Дополнительно у каждого объекта есть числовой суррогатный ключ для FK.

### 6. Аудит Действий

Все изменения, влияющие на данные (создание/удаление образцов, утверждение кариотипа, замена хромосомы), пишутся в `audit_log` с `actor`, `action`, `target_id`, `timestamp`, `payload`.

### 7. Сначала Работает, Потом Масштабируется

В v2 нагрузка маленькая: одна-две лаборатории, несколько пользователей, тысячи образцов в год. Не оптимизируем то, что не страдает. Пагинация, индексы, кеши — по мере необходимости.

## Стек

| Слой | Выбор | Альтернативы (если откажемся) |
|---|---|---|
| Язык | TypeScript (Node.js) | Python (FastAPI), Go |
| Фреймворк | Fastify или Express | Koa, Hono |
| ORM/Query Builder | Prisma или Kysely | Knex, Drizzle |
| БД | PostgreSQL | MySQL, SQLite (для локальной установки) |
| Хранилище медиа | Локальная файловая система + объектное (S3-совместимое) для prod | Только локальное на простой установке |
| Очередь задач (для PSD-парсинга, экспорта) | BullMQ + Redis | Inline-обработка, если объём небольшой |
| Логи | Pino + ротация | Winston |
| Тесты | Vitest или Jest | — |

Окончательный выбор фиксируется в [01_архитектура.md](01_архитектура.md) перед началом реализации.
